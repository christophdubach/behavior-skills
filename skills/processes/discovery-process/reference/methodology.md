## Problem Framing

A good problem statement:
- Names the affected user or system
- Describes the gap between current and desired state
- Avoids prescribing a solution

**Template:** "When [context], [actor] cannot [goal], which causes [impact]."

## Constraint Categories

| Category | Examples |
|---|---|
| Technical | Legacy systems, API limits, platform constraints |
| Time | Deadlines, release cycles |
| Resources | Budget, team size, skills |
| Political | Stakeholder dependencies, approval chains |
| Legal / compliance | GDPR, WCAG, brand rules |

## Assumption Risk Assessment

Rate each assumption: **high / medium / low** risk if wrong.  
High-risk assumptions should be validated before work begins.
